## 天翼云盘（Cloud189）实现文档

### 1. 文档目的

本文档基于当前项目源码整理，说明天翼云盘能力的实际实现方式，重点补全以下内容：

- 项目内天翼云盘模块的分层结构
- 登录认证与签名机制
- 后端 HTTP 接口清单
- SDK 内部核心方法与对应官方接口
- 自动转存任务的调用链
- 当前代码中的已知问题与注意事项

本文档对应的核心源码位置：

- `backend/api/cloud189.py`
- `backend/utils/cloud189/client.py`
- `backend/utils/cloud189/auth.py`
- `backend/utils/cloud189/const.py`
- `backend/utils/cloud189/util.py`
- `backend/utils/cloud189/types.py`
- `backend/task/cloud189_auto_save.py`

### 2. 模块分层

项目中的天翼云盘功能分为三层：

1. `backend/utils/cloud189/`
   这是核心 SDK，负责直接调用天翼云盘接口，处理登录、签名、会话、文件操作、分享转存等能力。
2. `backend/api/cloud189.py`
   这是 FastAPI 接口层，把 SDK 包装成项目内部可调用的 HTTP API。
3. `backend/task/cloud189_auto_save.py`
   这是自动化任务层，基于 SDK 实现“从 TG 资源消息中识别天翼云盘链接并自动保存”的业务流程。

调用关系如下：

- 前端调用 `/api/v1/cloud189/*`
- API 层通过 `Depends(get_client)` 获取 `Cloud189Client`
- `Cloud189Client` 调用天翼云盘官方接口
- 定时任务不经过 HTTP 层，直接实例化 `Cloud189Client`

### 3. 路由与鉴权

后端总路由前缀来自 `backend/main.py` 中的 `settings.app_config["API_V1_STR"]`，项目实际使用路径为：

- `/api/v1/cloud189/*`

同时，`/api/v1/cloud189` 被 `AuthCodeMiddleware` 保护，属于受保护接口。

此外，`backend/api/cloud189.py` 中的所有接口还依赖：

- `current_user: User = Depends(get_current_user)`

也就是说，调用这些接口通常需要：

- 通过项目自身登录体系拿到用户身份
- 满足中间件要求的授权码或鉴权条件

### 4. 配置项

#### 4.1 实际使用的配置键

天翼云盘主要读取以下配置：

- `tianyiAccount`
- `tianyiPassword`
- `tianyiCookie`
- `cloud189_session`

其中：

- `tianyiAccount` / `tianyiPassword` 是账号密码登录
- `tianyiCookie` 是 SSO Cookie 登录
- `cloud189_session` 是登录后缓存的会话，包含：
  - `access_token`
  - `session_key`
  - `expires_in`

#### 4.2 会话缓存机制

`Cloud189Client.login()` 的逻辑是：

1. 优先从配置中读取 `cloud189_session`
2. 如果 `expires_in` 未过期，则尝试直接用该会话初始化
3. 如果会话失效，再重新走登录流程
4. 登录成功后，把新会话写回配置

#### 4.3 已知配置键不一致

`backend/api/cloud189.py` 中存在一个明显不一致：

- `get_client()` 使用的是 `tianyiAccount` / `tianyiPassword`
- `/cloud189/init` 使用的是 `cloud189Account` / `cloud189Password`

但系统配置模型 `backend/schemas/sysSetting.py` 实际定义的是：

- `tianyiAccount`
- `tianyiPassword`

因此当前代码下：

- 大部分接口依赖 `get_client()`，应配置 `tianyiAccount` / `tianyiPassword`
- `/cloud189/init` 的配置读取逻辑与系统配置模型不一致，文档上应视为现有问题

### 5. 认证与签名机制

#### 5.1 常量

`backend/utils/cloud189/const.py` 中定义了三个关键域名：

- `WEB_URL = https://cloud.189.cn`
- `AUTH_URL = https://open.e.189.cn`
- `API_URL = https://api.cloud.189.cn`

#### 5.2 登录流程

`backend/utils/cloud189/auth.py` 的 `CloudAuthClient` 负责登录。

账号密码登录 `login_by_password()` 的流程：

1. 调用 `POST https://open.e.189.cn/api/logbox/config/encryptConf.do`
   获取 RSA 公钥和前缀
2. 调用 `GET https://cloud.189.cn/api/portal/unifyLoginForPC.action`
   获取登录页参数：
   - `captchaToken`
   - `lt`
   - `paramId`
   - `reqId`
3. 使用 `rsa_encrypt()` 对用户名和密码做 RSA 加密
4. 调用 `POST https://open.e.189.cn/api/logbox/oauth2/loginSubmit.do`
   提交加密后的登录表单
5. 从返回的 `toUrl` 换取 PC 会话
6. 调用 `POST https://api.cloud.189.cn/getSessionForPC.action`
   获取：
   - `sessionKey`
   - `accessToken`
   - `refreshToken`
   等会话字段

SSO Cookie 登录 `login_by_sson_cookie()` 的流程：

1. 先访问 `unifyLoginForPC.action`
2. 携带 `SSON=<cookie>` 请求重定向地址
3. 再调用 `getSessionForPC.action` 换取会话

#### 5.3 签名规则

`backend/utils/cloud189/util.py` 中的签名逻辑为：

1. 将请求参数整理为 `key=value`
2. 按字符串排序
3. 用 `&` 拼接
4. 对结果计算 MD5

对应方法：

- `sort_params(params)`
- `get_signature(params)`

#### 5.4 请求头附加规则

`Cloud189Client._send_request()` 会自动补充：

- `sessionKey`
- `noCache`
- `Timestamp`
- `Signature`
- `Accesstoken`

其中：

- 调用 `API_URL` 时，签名数据主要基于请求参数或表单，再加上 `Timestamp` 和 `AccessToken`
- 调用 `WEB_URL` 下 `/open/*` 接口时，会额外使用 `AppKey=600100422` 参与签名

#### 5.5 Token 失效自动恢复

`_send_request()` 会识别以下错误码：

- `InvalidAccessToken`
- `InvalidSessionKey`

遇到后会自动重新 `login()`，然后重试原请求。

### 8. 后端 HTTP 接口清单

以下接口定义于 `backend/api/cloud189.py`，统一前缀为：

- `/api/v1/cloud189`

返回格式分两类：

1. 使用项目统一包装：
   - `{"code": 200, "message": "操作成功", "data": ...}`
2. 直接返回原始对象或字符串

#### 8.1 `POST /cloud189/init`

用途：

- 初始化并登录 `Cloud189Client`
- 把客户端写入全局缓存 `client_cache`

请求体：

无

依赖配置：

- 当前代码读取 `cloud189Account`
- 当前代码读取 `cloud189Password`

返回：

- 成功时返回包含 `message` 和 `user_info`
- 失败时抛出 `HTTPException`

注意：

- 该接口读取的配置键与其他接口不一致

#### 8.2 `POST /cloud189/files`

用途：

- 列出个人网盘目录

请求体：

```json
{
  "folder_id": "-11"
}
```

字段说明：

- `folder_id`: 目录 ID，默认 `-11`，表示根目录

返回：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "fileListAO": {
      "folderList": [],
      "fileList": []
    }
  }
}
```

底层调用：

- `client.list_files(folder_id)`

#### 8.3 `POST /cloud189/search`

用途：

- 搜索个人网盘中的文件

请求体：

```json
{
  "keyword": "示例文件名"
}
```

返回：

- 直接返回文件列表数组

底层调用：

- `client.search_files(keyword)`

注意：

- 该接口没有使用统一 `Response` 包装

#### 8.4 `POST /cloud189/download`

用途：

- 获取文件下载直链

参数：

- `file_id: str`
- `share_id: str | null`

说明：

- 该接口函数参数不是 Pydantic 请求体模型，更接近 query/form 参数风格
- `share_id` 有值时按分享文件处理，否则按个人云盘文件处理

返回：

- 直接返回下载 URL 字符串

底层调用：

- `client.get_download_url(file_id, share_id)`

#### 8.5 `POST /cloud189/folder`

用途：

请求体：

返回：

- 统一 `Response` 包装

底层调用：

- `client.create_folder(folder_name, parent_id)`

#### 8.6 `POST /cloud189/rename`

用途：

- 重命名文件

请求体：

```json

  "file_id": "123456",
```

返回：

- 统一 `Response` 包装

底层调用：

#### 8.7 `POST /cloud189/share/info`

用途：

- 解析分享链接
- 可选校验访问码
- 返回分享信息

请求体：

```json
{
  "share_url": "https://cloud.189.cn/web/share?code=xxxx",
  "access_code": "abcd"
}
```

处理流程：

1. `parse_cloud_share()` 提取 URL 与访问码
2. `parse_share_code()` 提取分享码
3. 如果有访问码，先 `check_access_code()`
4. 调用 `get_share_info()`

返回：

- 直接返回分享信息对象 `ShareInfo`

#### 8.8 `POST /cloud189/share/files`

用途：

- 获取分享链接下的目录内容

请求体：

```json
{
  "share_url": "https://cloud.189.cn/web/share?code=xxxx",
  "access_code": "abcd",
  "file_id": "目录ID"
}
```

字段说明：

- `share_url`: 分享链接文本
- `access_code`: 访问码，可选
- `file_id`: 指定查看的分享目录 ID，可选；不传时默认使用分享根目录

处理流程：

1. 解析分享链接
2. 获取分享信息
3. 调用 `list_share_files(...)`

返回：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "fileListAO": {
      "folderList": [],
      "fileList": []
    }
  }
}
```

注意：

- 当前实现中这里对 `parse_cloud_share()` 的返回值使用方式不严谨，源码把它当成单个 URL 使用，但该方法实际返回 `(url, access_code)` 元组

#### 8.9 `POST /cloud189/share/save`

用途：

- 将分享文件保存到个人网盘目录

请求体：

```json
{
  "share_url": "https://cloud.189.cn/web/share?code=xxxx",
```

字段说明：

- `share_url`: 分享链接
- `target_folder_id`: 目标目录 ID，默认 `-11`
- `file_ids`: 需要保存的文件/目录列表；不传则由 SDK 侧按空列表处理

返回：

- 统一 `Response` 包装，`data` 为批量转存结果

- `client.save_share_files(share_url, target_folder_id, file_ids)`

- `message`
- `task_id`
- `status`
- `failed_files`
- `failed_count`

### 9. 分享解析能力

`Cloud189Client` 提供两个非常关键的静态方法。

#### 9.1 `parse_cloud_share(share_text)`

作用：

- 从一段混合文本中提取分享链接和访问码

返回：

- `(url, access_code)`

支持的访问码文案包括：

- `访问码：abcd`
- `提取码：abcd`
- `（访问码：abcd）`
- `（提取码：abcd）`

支持的分享链接格式包括：

- `https://cloud.189.cn/web/share?...`
- `https://cloud.189.cn/t/xxxx`
- `https://h5.cloud.189.cn/share.html#/t/xxxx`
- 其他域名下的 `/web/share`
- 其他域名下的 `/t/xxxx`
- `content.21cn.com` 订阅链接

#### 9.2 `parse_share_code(share_link)`

作用：

- 从分享链接中提取真正的分享码

支持格式：

- `/web/share?code=xxxx`
- `/t/xxxx`
- `share.html#/t/xxxx`
- `content.21cn.com` 订阅链接中的 `shareCode`

### 10. 转存与删除任务机制

天翼云盘项目中的“保存分享文件”和“删除文件”都不是一次性直接完成，而是通过批量任务接口执行。

#### 10.1 转存任务 `save_share_files()`

主要流程：

1. 先获取 `share_info`
2. 构造 `taskInfos`
3. 调用 `create_batch_task()` 创建 `SHARE_SAVE` 任务
4. 轮询 `check_task_status()`
5. 如果出现冲突状态 `taskStatus == 2`
   - 调用 `get_conflict_task_info()`
   - 把所有冲突项的 `dealWay` 设为 `1`
   - 即“跳过冲突”
   - 再调用 `manage_batch_task()` 继续执行
6. 任务完成后返回最终结果

当前冲突处理策略：

- `dealWay = 1`：跳过

也就是说，当前实现不会自动覆盖，也不会自动保存副本。

#### 10.2 删除任务 `delete_files()`

主要流程：

1. 构造 `DELETE` 类型批量任务
2. 调用 `create_batch_task()`
3. 轮询 `check_task_status(taskId, "DELETE")`
4. 如果 `taskStatus == 4` 视为完成
5. 如果返回里有失败计数，会尝试通过搜索文件名判断哪些文件仍然存在

### 11. 自动转存任务实现

`backend/task/cloud189_auto_save.py` 实现的是业务层自动化流程，不走 HTTP 接口。

#### 11.1 任务入口

主方法：

- `Cloud189AutoSave.cloud189_auto_save(task)`

依赖的任务参数主要包括：

- `shareUrl`
- `targetDir`
- `sourceDir`
- `replace`
- `renameStyle`
- `renameTemplate`
- `startMagic`
- `ignoreExtension`
- `search_pattern`

#### 11.2 执行流程

1. 从系统配置初始化 `Cloud189Client`
2. 登录天翼云盘
3. 解析 `shareUrl`
4. 获取 `share_info`
5. 调用 `dir_check_and_save()` 递归处理目录
6. 对目标目录做存在性检查
7. 过滤不需要保存的文件
8. 保存分享文件
9. 对保存后的文件按命名模板重命名
10. 发送任务开始、完成、异常通知

#### 11.3 目录递归处理

`dir_check_and_save()` 会：

- 读取分享目录内容
- 读取目标目录内容
- 对不存在的子目录自动创建
- 对文件做过滤和去重
- 调用 `_save_files()` 执行保存
- 调用 `_rename_saved_files()` 执行重命名

### 12. 前端实际使用到的接口

前端 `frontend/apps/web-antd/src/views/components/yunpanSave/api.ts` 当前使用的是：

- `POST /cloud189/files`
- `POST /cloud189/share/files`

- `POST /cloud189/folder`
- `POST /cloud189/rename`

```http
      "fileId": "123456",
      "fileName": "Movie.mkv",
      "isFolder": 0
    }
  ]
}
```

### 14. 已知问题与维护建议

#### 14.1 配置键不一致

如前所述：

- `get_client()` 使用 `tianyiAccount` / `tianyiPassword`
- `/cloud189/init` 使用 `cloud189Account` / `cloud189Password`

建议统一成一套配置键。

#### 14.2 `/share/files` 中对 `parse_cloud_share()` 的调用方式不严谨

`parse_cloud_share()` 返回的是元组：

- `(url, access_code)`

但 `/cloud189/share/files` 里把返回值直接赋给 `url` 使用。虽然 `parse_share_code()` 内部兼容了 tuple，但这里的写法可读性差，也容易诱发后续 bug。

建议改为：

```python
url, code = client.parse_cloud_share(req.share_url)
```

#### 14.3 返回结构不统一

当前接口有三种返回风格：

- `Response(data=...)`
- 直接返回数组
- 直接返回字符串

建议统一为项目标准响应结构，便于前端和第三方调用方处理。

#### 14.4 超时阈值偏短

`save_share_files()` 和 `delete_files()` 中轮询超时目前只有约 5 秒。

对于大文件或大批量任务，这个阈值偏短，建议改成可配置，例如：

- 30 秒
- 60 秒
- 或按任务大小动态调整

#### 14.5 `types.py` 中字段定义与接口原始字段并不完全一致

例如代码里实际读取的很多字段更接近：

- `id`
- `name`

但类型定义中同时存在另一套偏转换后的命名。

这不影响当前运行，但会降低类型提示的准确性，建议按真实响应结构整理一次类型模型。

### 15. 总结

本项目的天翼云盘实现已经具备完整闭环：

- 能登录
- 能缓存会话
- 能列目录
- 能解析分享
- 能保存分享文件
- 能创建目录、重命名、删除
- 能被前端界面和定时任务共同复用

如果后续继续维护，优先建议处理三件事：

1. 统一配置键与返回结构
2. 修正 `/share/files` 的参数解析写法
3. 把批量任务超时和冲突策略做成可配置
