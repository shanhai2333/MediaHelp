## 天翼云盘（Cloud189）功能实现文档

### 1. 总体架构

本项目中的天翼云盘功能主要由三个核心部分组成：

1.  **核心SDK (`backend/utils/cloud189/`)**: 一个独立的工具包，封装了与天翼云盘官方API交互的所有底层逻辑，包括认证、签名、请求、文件操作等。这是整个功能的基础。
2.  **API接口层 (`backend/api/cloud189.py`)**: 使用 FastAPI 框架，将核心SDK的功能封装成HTTP接口，供前端或其他服务调用。
3.  **后台任务 (`backend/task/cloud189_auto_save.py`)**: 一个示例性的后台定时任务，演示了如何利用核心SDK实现自动化操作（如此处的从Telegram自动转存）。

三者的关系是：**API接口层** 和 **后台任务** 都调用 **核心SDK** 来完成具体的天翼云盘操作。

### 2. 核心SDK (`backend/utils/cloud189/`)

这是与天翼云盘交互的 Python 客户端，设计成可以独立使用。

#### a. `client.py` - 客户端主文件

这是SDK的核心，定义了 `Cloud189Client` 类。

*   **主要职责**: 封装所有对天翼云盘API的请求。
*   **初始化**: `__init__(self, username, password, sson_cookie)` 接收多种认证方式。
*   **登录与会话**:
    *   `login()`: 自动处理登录逻辑。它会先尝试从配置文件加载并验证现有会话，如果会话失效，则根据提供的 `username`/`password` 或 `sson_cookie` 重新登录。
    *   登录成功后，`session` (包含 `access_token` 和 `session_key`) 会被缓存并保存到配置文件，实现持久化登录。
*   **请求签名**:
    *   `_send_request()`: 内部私有方法，所有API请求都通过它发送。
    *   它会自动在请求中加入 `sessionKey`、时间戳、`noCache`随机数等。
    *   最关键的是，它会调用 `util.py` 中的 `get_signature()` 方法对请求参数进行签名，并将签名附加到请求头中，这是通过天翼云盘API认证的关键。
*   **Token自动刷新**: `_send_request()` 会捕获 `InvalidAccessToken` 和 `InvalidSessionKey` 错误，并在捕获到错误时自动调用 `login()` 方法刷新Token，然后重试失败的请求，大大提高了健壮性。
*   **主要功能方法**:
    *   `get_user_info()`: 获取用户信息和容量。
    *   `list_files()`: 列出指定目录的文件。
    *   `save_share_files()`: **核心功能**，用于转存分享链接中的文件。它会自动处理文件冲突（默认跳过）。
    *   `delete_files()`: 删除文件。
    *   `create_folder()`, `rename_file()`: 创建和重命名文件夹。
    *   `get_download_url()`: 获取文件下载直链。
    *   `parse_cloud_share()`: 从混杂的文本中智能提取分享链接和访问码。

#### b. `auth.py` - 认证模块

定义了 `CloudAuthClient` 类，专门处理复杂的登录认证流程。

*   **主要职责**: 执行登录操作并返回有效的会话凭证。
*   **密码登录流程 (`login_by_password`)**:
    1.  获取加密公钥 (`get_encrypt_conf`)。
    2.  使用公钥对用户名和密码进行 **RSA加密** (`_build_login_form`)。
    3.  提交加密后的表单进行登录。
    4.  用登录成功后返回的 `toUrl` 获取最终的 `session`。
*   **其他登录方式**: 还支持 `login_by_sson_cookie` (SSO单点登录) 和 `refresh_token` (刷新令牌)。

#### c. `util.py` - 工具函数

提供SDK所需的通用工具。

*   `rsa_encrypt(public_key, data)`: 使用 `pycryptodome` 库，根据天翼云盘的公钥对数据进行 **PKCS1_v1_5** 模式的RSA加密。
*   `get_signature(params)`: 对所有请求参数按key排序，拼接成 `key=value` 字符串，然后计算其 **MD5值** 作为签名。

#### d. `types.py` - 数据类型定义

使用 `TypedDict` 定义了所有与API交互的数据结构，如 `FileInfo`, `ShareInfo`, `TokenSession` 等，增强了代码的可读性和健壮性。

### 3. API接口层 (`backend/api/cloud189.py`)

这一层是连接前端和后端SDK的桥梁。

*   **框架**: 使用 `FastAPI` 创建RESTful API。
*   **依赖注入 (`get_client`)**:
    *   定义了一个 `get_client()` 函数，作为FastAPI的依赖项。
    *   这个函数负责从系统配置中读取天翼云盘的用户名和密码，创建并缓存 `Cloud189Client` 的全局单例。
    *   这意味着整个应用共享同一个客户端实例，避免了重复登录。如果实例未创建或登录失败，会直接向前端返回HTTP错误。
*   **接口定义**:
    *   使用 `@router.post(...)` 等装饰器定义HTTP接口。
    *   每个接口都接收一个通过 `Depends(get_client)` 注入的 `client` 实例。
    *   接口函数内部只需调用 `client` 实例对应的方法即可，例如 `list_files` 接口调用 `client.list_files()`。
    *   使用 `Pydantic` 模型（如 `FileListRequest`）来验证和解析请求体。

### 4. 后台任务 (`backend/task/cloud189_auto_save.py`)

这是一个具体的业务应用，展示了如何调用SDK实现自动化。

*   **功能**: 自动将Telegram消息中的天翼云盘链接保存到指定的云盘目录。
*   **`auto_save_from_tg(tg_channel_id, save_dir_id)`**:
    *   从数据库读取指定Telegram频道的消息。
    *   调用 `client.parse_cloud_share()` 从消息文本中提取链接和访问码。
    *   调用 `client.save_share_files()` 将文件保存到指定的 `save_dir_id` 目录。
    *   处理成功后更新数据库状态，避免重复处理。
*   **触发方式**: 这个函数被一个调度器（`scheduler_manager`）注册为一个定时任务，可以按设定的周期（例如每30分钟）自动执行。

### 5. 总结与借鉴建议

*   **分层设计**: 该项目优秀地实践了分层设计。`utils/cloud189` 是一个高度内聚、可复用的SDK，你可以直接将这个目录复制到你的项目中。`api` 层则清晰地展示了如何将SDK能力服务化。
*   **认证和签名是关键**: 如果你要自己实现，重点是搞清楚 `auth.py` 的RSA加密和 `util.py` 的MD5签名逻辑，这是与天翼云盘API通信的两个关键点。
*   **错误处理和Token刷新**: `Cloud189Client` 中的 `_send_request` 方法里对Token失效的自动处理机制非常值得学习，能让你的应用更加稳定。
*   **从解析分享链接开始**: `client.py` 中的 `parse_cloud_share` 和 `parse_share_code` 两个静态方法非常实用，可以帮你轻松地从各种格式的分享文案中提取出有效信息。
